# context.md

# 少年野球向け初心者用スコアブックPWA コンテキスト

## 1. アプリの目的

少年野球の試合中に、野球やスコアブックに詳しくない保護者でも、スマホで簡単に記録できるスコアブックPWAを作る。

対象ユーザーは、少年野球の母・父など、野球ルールや早稲田式スコアブックの記録方法に詳しくない人。

最優先は、正確な公式記録を作ることではなく、試合中に迷わず、止まらず、あとから見返せる記録を残すこと。

## 2. 最重要コンセプト

このアプリは「野球に詳しい人が細かく記録するアプリ」ではない。

野球に詳しくない人でも、見えた事実だけを入れれば記録が進むアプリにする。

最重要ルールは以下。

```text
ユーザーに野球ルールの判断を強要しない。
ユーザーには、見えた事実だけを入力させる。
不明な場合は常に「？」を選べる。
入力矛盾があっても保存と進行を許可する。
ただし、未確定・矛盾のある箇所には要確認マークを付ける。
あとから状況が確定したら、過去の「？」を補正できる。
```

## 3. ことばの方針

野球初心者、特に少年野球の保護者に馴染みやすいことばを使う。

UI上では、難しい用語や公式記録寄りの用語を避ける。

例。

- 「打者」ではなく「バッター」
- 「走者」ではなく「ランナー」
- 「投手」ではなく「ピッチャー」
- 「打席」よりも必要に応じて「このバッター」
- 「打順」は使ってよい
- 「アウト」「セーフ」「ヒット」「フォアボール」「三振」は使ってよい

内部コードでは `batter`, `runner`, `pitcher` などの英語名を使ってよいが、画面表示は初心者向けの日本語にする。

## 4. MVPの方針

MVPでは、とにかくシンプルにする。

選手管理アプリにしない。  
詳細成績アプリにしない。  
事前準備を重くしない。  
まずは「試合中に入力できるスコアブックアプリ」にする。

MVPで重視すること。

- スマホで試合中に使いやすい
- 情報を詰め込みすぎない
- ボタンの大きさそのものにはこだわらない
- 大きさよりも最適な配置と操作の迷わなさを優先する
- 入力項目が少ない
- わからないときに「？」で進められる
- 間違えても1球ずつ戻せる
- 球数制限のため、ピッチャーごとの球数がわかる
- 保存済み試合をあとから見返せる
- バックアップ/復元できる

## 5. 技術方針

- PWAとして実装する
- React + TypeScript を使用する
- Vite を使用する
- スマホ優先UI
- ローカル保存のみ
- IndexedDBを使用する
- Dexie.jsを使用してよい
- PDF出力はまず印刷用CSS + `window.print()` でよい
- オフラインで使えること
- クラウド同期、ログイン、複数端末共有はMVPでは不要

## 6. バックアップ方針

バックアップはCSVにこだわらない。

IndexedDBの内容をそのままエクスポートして、あとでインポートできる形式でよい。

MVPでは以下を満たせばよい。

- 全データをファイルとしてエクスポートできる
- エクスポートしたファイルをインポートして復元できる
- 形式はJSONでもよい
- Dexie/IndexedDBの内容をアプリ独自JSONとして保存してよい

CSV分析用途はMVPでは不要。

## 7. データ保存の最重要方針

スコアブックの見た目を直接保存しない。

必ずイベントログを保存する。

イベントログから以下を再計算する。

- 現在の回
- 表/裏
- 打順
- アウト数
- ランナー状況
- 得点
- ピッチャー球数
- スコアブック風表示
- 要確認リスト
- 勉強モードの説明文

重要。

```text
表示結果を保存するのではなく、入力イベントを保存する。
修正時はイベントログを変更し、その時点以降の状態を再計算する。
```

## 8. 画面構成

MVPの画面は以下に絞る。

1. ホーム
2. 自チーム選手
3. 試合作成
4. 試合入力
5. 閲覧
6. 勉強モード

事前のオーダー入力画面は作らない。  
打順は試合中、初打席のときに決める。

## 9. ホーム画面

表示するもの。

- 新しい試合
- 保存済み試合
- 自チーム選手
- バックアップ
- 復元

## 10. 自チーム選手

自チーム選手は恒久的に保存し、毎試合使い回せる。

MVPの選手情報はこれだけ。

- 背番号
- 名前

不要な項目。

- よみがな
- 投打
- 守備位置候補
- 表示順
- 有効/無効
- 詳細プロフィール

表示例。

```text
1　山田
3　佐藤
5　田中
7　鈴木
10　高橋
```

操作はシンプルにする。

- 選手を追加
- 編集
- 削除

## 11. 相手チーム選手

相手チームは事前登録しない。  
相手チームのオーダーも事前入力しない。

相手チームのバッターが初めて出てきたときに、背番号を入力させる。

背番号がわからないときは「？」でよい。  
次の打席でもまた聞く。  
あとから修正できるようにする。

表示例。

```text
相手の1番バッターの背番号は？
[  ]  /  ？
```

以降は、入力済みなら以下のように表示する。

```text
相手 1番：背番号7
```

わからない場合。

```text
相手 1番：？
```

## 12. 試合作成

MVPの入力項目は以下だけ。

- 試合日
- 大会名
- 自チーム名
- 相手チーム名
- 自チームが先攻か後攻か
- 球数上限

球場、イニング数、警告ラインなどはMVPでは不要。

球数警告は、球数上限の10球前で自動表示する。

例。

```text
球数上限: 70球
警告: 60球
上限: 70球
```

## 13. 打順の扱い

### 13.1 自チームの打順

自チームの打順は、試合前に事前入力しない。

初打席のときに選手リストから選ばせる。

例。

```text
自チームの1番バッターを選んでください
[山田]
[佐藤]
[田中]
[鈴木]
```

一度選んだら、その試合中の打順として保存する。

次に同じ打順が回ってきたら、自動でその選手を表示する。

例。

```text
1番バッター：山田
```

### 13.2 相手チームの打順

相手チームの打順も、試合前に事前入力しない。

初打席のときに背番号を入力させる。

例。

```text
相手の1番バッターの背番号は？
[  ]  /  ？
```

一度入力したら、その試合中の打順として保存する。

次に同じ打順が回ってきたら、自動で表示する。

例。

```text
相手 1番バッター：背番号7
```

不明の場合は「？」のままでよい。

## 14. 選手交代

バッターが変わった場合は「交代」ボタンを押す。

### 14.1 自チームの交代

自チームの場合は、登録済み選手リストから代わりの選手を選ぶ。

例。

```text
交代する選手を選んでください
[山田]
[佐藤]
[田中]
[鈴木]
```

選んだ選手を、その打順の新しいバッターとして扱う。

### 14.2 相手チームの交代

相手チームの場合は、代わりの選手の背番号を入力する。

例。

```text
代わりのバッターの背番号は？
[  ]  /  ？
```

不明なら「？」でよい。

### 14.3 交代の詳細

MVPでは交代の詳細ルールは扱わない。

- 守備交代
- 代打
- 代走
- 再出場
- 守備位置変更

これらはMVPでは厳密に扱わない。  
「その打順のバッターが変わった」だけを記録する。

## 15. 試合入力画面

試合中に使うメイン画面。

スマホで見やすく、操作の迷わなさを優先する。

ボタンは必ずしも大きくなくてよい。  
画面全体の最適配置、誤操作しにくさ、入力の流れを優先する。

### 15.1 常時表示する情報

画面上部に現在状況を表示する。

例。

```text
3回表　1アウト
自チーム 2 - 1 相手

バッター：山田
ピッチャー：背番号11　42球

一塁：背番号7
二塁：なし
三塁：佐藤
```

表示する情報。

- 回
- 表/裏
- アウト数
- スコア
- 現在のバッター
- 現在のピッチャー
- 現在ピッチャーの球数
- 一塁
- 二塁
- 三塁

### 15.2 投球入力

球数制限があるため、投球入力はMVPに入れる。

ボタン。

```text
S
B
F
？
```

意味。

- S = ストライク
- B = ボール
- F = ファール
- ? = 判定不明

ルール。

- どのボタンでも球数は1増える
- 「？」も球数に含める
- ピッチャーごとに球数をカウントする
- ストライク判定を聞き逃しても「？」で進められる

### 15.3 結果入力

ボタン。

```text
ヒット
アウト
三振
フォアボール
エラーっぽい
何か起きた
？
交代
```

MVPでは結果を細かく分けすぎない。

「何か起きた」は、ボーク、インフィールドフライ、タッチアップ、野選、ワイルドピッチ、パスボール、走塁アウトなど、初心者が判断しにくいプレーをまとめて扱う入口にする。

### 15.4 補助ボタン

常時表示する補助ボタンは以下。

```text
戻す
状況確認
あとで確認
```

「戻す」は1球ずつ戻す。  
何回も押せば何球でも戻れる。

「点が入った」「チェンジになった」「誰がどこにいるか」は、常時ボタンではなく「状況確認」にまとめる。

## 16. ヒット入力

ヒットを押した場合は、まずバッターが何塁まで行ったかを聞く。

例。

```text
山田は何塁まで行きましたか？

一塁
二塁
三塁
ホーム
アウト
？
```

ランナーがいる場合は、それぞれのランナーが何塁まで行ったかを聞く。

例。

```text
佐藤は何塁まで行きましたか？

二塁
三塁
ホーム
アウト
動いていない
？
```

相手チームの場合は背番号で聞く。

```text
背番号7は何塁まで行きましたか？

二塁
三塁
ホーム
アウト
動いていない
？
```

ユーザーに「単打」「二塁打」「三塁打」などを先に判断させない。  
バッターがどこまで進んだかから、アプリ側で結果を推定する。

## 17. 状況確認

試合中に迷ったときに押すボタン。

「状況確認」を押すと、現在の状況をまとめて修正・確認できる。

### 17.1 状況確認で聞くこと

- アウト数
- このプレーで入った得点
- 各選手が今どこにいるか

例。

```text
いまの状況を教えてください

アウト数
0 / 1 / 2 / チェンジ / ？

このプレーで入った得点
0 / 1 / 2 / 3以上 / ？

山田
ベンチ / 一塁 / 二塁 / 三塁 / ホーム / アウト / ？

背番号7
ベンチ / 一塁 / 二塁 / 三塁 / ホーム / アウト / ？
```

## 18. 選手表示ルール

試合中のポップアップでは、塁ではなく選手を起点に聞く。

悪い例。

```text
一塁ランナーはどこまで進みましたか？
```

良い例。

```text
山田は今どこにいますか？
背番号7は今どこにいますか？
```

表示ルール。

- 自チーム選手は名前で表示する
- 相手チーム選手は背番号で表示する
- 相手チームの背番号がわからない場合は「？」で表示する

## 19. 専門用語を使わない方針

ユーザーに以下のような専門用語の判断を求めない。

- ボーク
- インフィールドフライ
- タッチアップ
- 犠牲フライ
- 野選
- 振り逃げ
- ワイルドピッチ
- パスボール
- 牽制アウト
- 走塁アウト

代わりに、見えた結果を聞く。

例。

```text
バッターはアウトになりましたか？
ランナーは進みましたか？
点は入りましたか？
誰がどこにいますか？
```

アウトかセーフかわからない場合も「？」で進められるようにする。

## 20. 「？」と矛盾の扱い

このアプリでは、不明や矛盾をエラーにしない。

### 20.1 「？」を許容する場面

- 投球内容が不明
- 結果が不明
- ランナーの行き先が不明
- アウト/セーフが不明
- 得点者が不明
- アウト数が不明
- チェンジかどうか不明
- 相手バッターの背番号が不明

### 20.2 矛盾を許容する例

以下のような場合でも保存・進行を許可する。

- 三振なのにストライク数が足りない
- フォアボールなのにボール数が足りない
- アウト数が合わない
- 得点が合わない
- 同じ塁に複数人がいる
- チェンジなのに記録上は2アウト
- 4アウト相当になっている
- 相手バッターの背番号が不明のまま進んでいる

ただし、要確認マークを付ける。

## 21. 三振・フォアボールからの補完

投球ログに「？」がある場合、結果から推測できるなら補完してよい。

例。

```text
投球ログ: ? / ? / S
結果: 三振
```

補完候補。

```text
S / S / S
```

例。

```text
投球ログ: B / ? / ? / B
結果: フォアボール
```

補完候補。

```text
B / B / B / B
```

ただし、完全に推測できない場合は「？」のまま残す。

補完した場合も、内部的には「推測済み」と分かるようにする。

## 22. あとで確認

「あとで確認」ボタンを押すと、現在のバッターまたはプレーに要確認マークを付ける。

閲覧画面で要確認リストを表示する。

例。

```text
要確認
・2回表 山田：ランナーの進塁が不明
・3回裏 背番号8：アウト/セーフ不明
・4回表：球数の内訳が不明
```

要確認項目をタップすると編集できるようにする。

## 23. 戻す

試合入力画面の「戻す」は1球ずつ戻す。

1回押すと、直前の1球を取り消す。  
何回も押せば、何球も戻せる。

打席結果まで戻したい場合も、イベントログを逆順に戻していく。

MVPでは「この打席をやり直す」「直前の打席を戻す」の専用ボタンは不要。

## 24. ピッチャー球数

投球入力ごとに現在のピッチャーの球数を1増やす。

ピッチャーごとの球数一覧を持つ。

表示例。

```text
ピッチャー別球数
山田 42球
佐藤 18球
```

相手チームピッチャーの場合。

```text
ピッチャー別球数
背番号11 42球
背番号3 18球
```

球数上限の10球前で注意表示する。

例。

```text
背番号11が60球に到達しました。
球数制限に注意してください。
```

上限に到達したら警告表示する。

```text
背番号11が70球に到達しました。
球数上限です。
```

ただし、入力は止めない。

## 25. 閲覧画面

保存済み試合を見直す画面。

表示するもの。

- スコアブック風表示
- イニングごとの得点
- ピッチャー別球数
- 要確認リスト
- PDF出力ボタン

MVPでは早稲田式の完全再現は不要。  
まずは早稲田式風に、あとから試合内容を追えることを優先する。

## 26. 勉強モード

閲覧モードで、特定のマスをタップすると、そのバッターで何が起きたかを日本語で説明する。

これは「勉強モード」と呼ぶ。

目的。

- 早稲田式の記号が読めない初心者でも内容を理解できるようにする
- 試合後に、記録の意味を学べるようにする
- 子どもや保護者が「このプレーは何だったのか」を見返せるようにする

例。

```text
2回表　山田

このバッターはヒットで一塁まで進みました。
一塁にいた佐藤は三塁まで進みました。
このプレーで1点入りました。
```

不明がある場合。

```text
3回裏　背番号8

このプレーは内容が一部わかっていません。
ランナーがどこまで進んだかが未確認です。
あとで確認が必要です。
```

矛盾がある場合。

```text
4回表　田中

記録上は2アウトですが、このあとチェンジになっています。
どこかでアウトが1つ記録されていない可能性があります。
```

## 27. PDF出力

閲覧画面からPDF出力できるようにする。

MVPではブラウザ印刷でよい。

実装方針。

- 印刷用CSSを用意する
- A4で見やすくする
- スコアブック風表示
- ピッチャー別球数
- 要確認リスト

を含める。

## 28. バックアップ/復元

ローカル保存は端末依存なので、バックアップできるようにする。

MVPではCSVである必要はない。

推奨。

- JSON形式で全データをエクスポート
- JSONファイルから全データをインポート
- IndexedDBのテーブル内容をアプリ独自形式で保存
- 復元時は既存データに追加するか、全置換するかを選べる

## 31. UIデザイン方針

- スマホで使いやすいこと
- ボタンの大きさよりも配置の最適化を優先すること
- 画面に情報を詰め込みすぎないこと
- 入力画面はシンプルにすること
- 補助操作は「戻す」「状況確認」「あとで確認」にまとめること
- 専門用語をなるべく使わないこと
- 不明な場合は必ず「？」を選べること
- エラーで入力を止めないこと
- 要確認は目立たせるが、試合中の邪魔をしすぎないこと
- 閲覧モードでは、マスをタップして日本語説明を見られるようにすること

## 32. 完成イメージ

最初の完成形は以下。

```text
少年野球の試合中に、保護者がスマホで
S / B / F / ?
ヒット / アウト / 三振 / フォアボール / エラーっぽい / 何か起きた / ?
を押していく。

自チームのバッターは、初打席のときに選手リストから選ぶ。
相手チームのバッターは、初打席のときに背番号を入力する。
わからなければ「？」で進める。

バッターが変わったら「交代」を押して、代わりの選手を選ぶ。
相手チームなら代わりの背番号を入力する。

ヒットの場合は、
バッターが何塁まで行ったか、
ランナーがいた場合はそのランナーが何塁まで行ったか、
を聞く。

迷ったら「状況確認」を押す。
状況確認では、アウト数、得点、選手が今どこにいるかをまとめて入力する。

自チーム選手は名前で表示する。
相手チーム選手は背番号で表示する。

わからない場合は「？」で進める。

アプリは得点、アウト数、ランナー、打順、ピッチャー球数を更新する。
矛盾や不明は要確認として残す。

試合後、閲覧画面でスコアブック風に見直せる。
マスをタップすると、そのバッターで何が起きたかを日本語で説明する。
PDF出力とJSONバックアップ/復元ができる。
```

## 33. Codexへの注意

まずは見た目よりも、以下を優先して実装する。

- イベントログ設計
- 状態再計算
- 「？」入力の許容
- 矛盾をエラーにしない設計
- 初打席時に打順を自然に埋める設計
- 1球ずつ戻せる設計
- 試合中に迷わないシンプルUI
- 閲覧モードの勉強モード

早稲田式スコアブックの見た目は、MVPでは簡易版でよい。
