# context.md

# 少年野球向け初心者用スコアブックPWA コンテキスト

## 1. アプリの目的

少年野球の試合中に、野球やスコアブックに詳しくない保護者でも、スマホで簡単に記録できるスコアブックPWAを作る。

対象ユーザーは、少年野球の母・父など、野球ルールや早稲田式スコアブックの記録方法に詳しくない人。

最優先は、正確な公式記録を作ることではなく、試合中に迷わず、止まらず、あとから見返せる記録を残すこと。

## 2. 最重要コンセプト

このアプリは「野球に詳しい人が細かく記録するアプリ」ではない。

野球に詳しくない人でも、見えた事実だけを入れれば記録が進むアプリにする。

最重要ルールは以下。

```text
ユーザーに野球ルールの判断を強要しない。
ユーザーには、見えた事実だけを入力させる。
不明な場合は常に「？」を選べる。
入力矛盾があっても保存と進行を許可する。
ただし、未確定・矛盾のある箇所には要確認マークを付ける。
あとから状況が確定したら、過去の「？」を補正できる。
```

## 3. MVPの方針

MVPでは、とにかくシンプルにする。

選手管理アプリにしない。  
詳細成績アプリにしない。  
まずは「試合中に入力できるスコアブックアプリ」にする。

MVPで重視すること。

- スマホで片手操作しやすい
- ボタンが大きい
- 入力項目が少ない
- わからないときに「？」で進められる
- 間違えても簡単に戻せる
- 球数制限のため、投手ごとの球数がわかる
- 保存済み試合をあとから見返せる
- PDF出力とCSVバックアップができる

## 4. 技術方針

- PWAとして実装する
- React + TypeScript を使用する
- Vite を使用する
- スマホ優先UI
- ローカル保存のみ
- IndexedDBを使用する
- Dexie.jsを使用してよい
- PDF出力はまず印刷用CSS + `window.print()` でよい
- CSVエクスポート/インポートに対応する
- オフラインで使えること
- クラウド同期、ログイン、複数端末共有はMVPでは不要

## 5. データ保存の最重要方針

スコアブックの見た目を直接保存しない。

必ずイベントログを保存する。

イベントログから以下を再計算する。

- 現在の回
- 表/裏
- 打順
- アウト数
- ランナー状況
- 得点
- 投手球数
- スコアブック風表示
- 要確認リスト

重要。

```text
表示結果を保存するのではなく、入力イベントを保存する。
修正時はイベントログを変更し、その時点以降の状態を再計算する。
```

## 6. 画面構成

MVPの画面は以下に絞る。

1. ホーム
2. 自チーム選手
3. 試合作成
4. オーダー入力
5. 試合入力
6. 閲覧

## 7. ホーム画面

表示するもの。

- 新しい試合
- 保存済み試合
- 自チーム選手
- CSVエクスポート
- CSVインポート

## 8. 自チーム選手

自チーム選手は恒久的に保存し、毎試合使い回せる。

MVPの選手情報はこれだけ。

- 背番号
- 名前

不要な項目。

- よみがな
- 投打
- 守備位置候補
- 表示順
- 有効/無効
- 詳細プロフィール

表示例。

```text
1　山田
3　佐藤
5　田中
7　鈴木
10　高橋
```

操作はシンプルにする。

- 選手を追加
- 編集
- 削除

## 9. 相手チーム選手

相手チームはその試合限りでよい。

MVPでは名前入力は不要。  
背番号だけで扱う。

相手チームのオーダー入力例。

```text
1番　背番号 [  ]
2番　背番号 [  ]
3番　背番号 [  ]
4番　背番号 [  ]
5番　背番号 [  ]
6番　背番号 [  ]
7番　背番号 [  ]
8番　背番号 [  ]
9番　背番号 [  ]
```

## 10. 試合作成

MVPの入力項目は以下だけ。

- 試合日
- 大会名
- 自チーム名
- 相手チーム名
- 自チームが先攻か後攻か
- 球数上限

球場、イニング数、警告ラインなどはMVPでは不要。

球数警告は、球数上限の10球前で自動表示する。

例。

```text
球数上限: 70球
警告: 60球
上限: 70球
```

## 12. 試合入力画面

試合中に使うメイン画面。

スマホで見やすく、大きなボタンを優先する。

### 12.1 常時表示する情報

画面上部に現在状況を表示する。

例。

```text
3回表　1アウト
自チーム 2 - 1 相手

打者：山田
投手：背番号11　42球

一塁：背番号7
二塁：なし
三塁：佐藤
```

表示する情報。

- 回
- 表/裏
- アウト数
- スコア
- 現在の打者
- 現在の投手
- 現在投手の球数
- 一塁
- 二塁
- 三塁

### 12.2 投球入力

球数制限があるため、投球入力はMVPに入れる。

ボタン。

```text
S
B
F
？
```

意味。

- S = ストライク
- B = ボール
- F = ファール
- ? = 判定不明

ルール。

- どのボタンでも投球数は1増える
- 「？」も球数に含める
- 投手ごとに球数をカウントする
- ストライク判定を聞き逃しても「？」で進められる

### 12.3 打席結果入力

ボタン。

```text
ヒット
アウト
三振
四球
エラーっぽい
何か起きた
？
```

MVPでは結果を細かく分けすぎない。

「何か起きた」は、ボーク、インフィールドフライ、タッチアップ、野選、暴投、捕逸、走塁死など、初心者が判断しにくいプレーをまとめて扱う入口にする。

### 12.4 補助ボタン

常時表示する補助ボタンは3つに絞る。

```text
戻す
状況確認
あとで確認
```

「点が入った」「チェンジになった」「誰がどこにいるか」は、常時ボタンではなく「状況確認」にまとめる。

## 13. 状況確認

試合中に迷ったときに押すボタン。

「状況確認」を押すと、現在の状況をまとめて修正・確認できる。

### 13.1 状況確認で聞くこと

- アウト数
- このプレーで入った得点
- 各選手が今どこにいるか

例。

```text
いまの状況を教えてください

アウト数
0 / 1 / 2 / チェンジ / ？

このプレーで入った得点
0 / 1 / 2 / 3以上 / ？

山田
ベンチ / 一塁 / 二塁 / 三塁 / ホーム / アウト / ？

背番号7
ベンチ / 一塁 / 二塁 / 三塁 / ホーム / アウト / ？
```

## 14. 選手表示ルール

試合中のポップアップでは、塁ではなく選手を起点に聞く。

悪い例。

```text
一塁走者はどこまで進みましたか？
```

良い例。

```text
山田は今どこにいますか？
背番号7は今どこにいますか？
```

表示ルール。

- 自チーム選手は名前で表示する
- 相手チーム選手は背番号で表示する
- 相手チームは名前入力を不要にする

## 15. 専門用語を使わない方針

ユーザーに以下のような専門用語の判断を求めない。

- ボーク
- インフィールドフライ
- タッチアップ
- 犠牲フライ
- 野選
- 振り逃げ
- 暴投
- 捕逸
- 牽制死
- 走塁死

代わりに、見えた結果を聞く。

例。

```text
打者はアウトになりましたか？
ランナーは進みましたか？
点は入りましたか？
誰がどこにいますか？
```

アウトかセーフかわからない場合も「？」で進められるようにする。

## 16. 「？」と矛盾の扱い

このアプリでは、不明や矛盾をエラーにしない。

### 16.1 「？」を許容する場面

- 投球内容が不明
- 打席結果が不明
- ランナーの行き先が不明
- アウト/セーフが不明
- 得点者が不明
- アウト数が不明
- チェンジかどうか不明

### 16.2 矛盾を許容する例

以下のような場合でも保存・進行を許可する。

- 三振なのにストライク数が足りない
- 四球なのにボール数が足りない
- アウト数が合わない
- 得点が合わない
- 同じ塁に複数人がいる
- チェンジなのに記録上は2アウト
- 4アウト相当になっている

ただし、要確認マークを付ける。

## 17. 三振・四球からの補完

投球ログに「？」がある場合、打席結果から推測できるなら補完してよい。

例。

```text
投球ログ: ? / ? / S
結果: 三振
```

補完候補。

```text
S / S / S
```

例。

```text
投球ログ: B / ? / ? / B
結果: 四球
```

補完候補。

```text
B / B / B / B
```

ただし、完全に推測できない場合は「？」のまま残す。

補完した場合も、内部的には「推測済み」と分かるようにする。

## 18. あとで確認

「あとで確認」ボタンを押すと、現在の打席またはプレーに要確認マークを付ける。

閲覧画面で要確認リストを表示する。

例。

```text
要確認
・2回表 山田：走者の進塁が不明
・3回裏 背番号8：アウト/セーフ不明
・4回表：球数の内訳が不明
```

要確認項目をタップすると編集できるようにする。

## 19. 戻す

試合入力画面の補助ボタンは「戻す」ひとつにまとめる。

押すと以下のメニューを出す。

```text
直前の1球を戻す
この打席をやり直す
直前の打席を戻す
```

## 20. 投手球数

投球入力ごとに現在の投手の球数を1増やす。

投手ごとの球数一覧を持つ。

表示例。

```text
投手別球数
山田 42球
佐藤 18球
```

相手チーム投手の場合。

```text
投手別球数
背番号11 42球
背番号3 18球
```

球数上限の10球前で注意表示する。

例。

```text
背番号11が60球に到達しました。
球数制限に注意してください。
```

上限に到達したら警告表示する。

```text
背番号11が70球に到達しました。
球数上限です。
```

ただし、入力は止めない。

## 21. 閲覧画面

保存済み試合を見直す画面。

表示するもの。

- スコアブック風表示
- イニングごとの得点
- 投手別球数
- 要確認リスト
- PDF出力ボタン

MVPでは早稲田式の完全再現は不要。  
まずは早稲田式風に、あとから試合内容を追えることを優先する。

## 22. PDF出力

閲覧画面からPDF出力できるようにする。

MVPではブラウザ印刷でよい。

実装方針。

- 印刷用CSSを用意する
- A4で見やすくする
- スコアブック風表示
- 投手別球数
- 要確認リスト

を含める。

## 23. CSVエクスポート/インポート

ローカル保存は端末依存なので、CSVでバックアップできるようにする。

MVPで必要なCSV。

- 自チーム選手
- 試合一覧
- 試合イベントログ

CSVの分析用途は後回し。  
まずはバックアップと復元を優先する。

## 26. UIデザイン方針

- スマホで使いやすいこと
- ボタンを大きくすること
- 画面に情報を詰め込みすぎないこと
- 入力画面はシンプルにすること
- 補助操作は「戻す」「状況確認」「あとで確認」にまとめること
- 専門用語をなるべく使わないこと
- 不明な場合は必ず「？」を選べること
- エラーで入力を止めないこと
- 要確認は目立たせるが、試合中の邪魔をしすぎないこと

## 27. 完成イメージ

最初の完成形は以下。

```text
少年野球の試合中に、保護者がスマホで
S / B / F / ?
ヒット / アウト / 三振 / 四球 / エラーっぽい / 何か起きた / ?
を押していく。

迷ったら「状況確認」を押す。

状況確認では、
アウト数、得点、選手が今どこにいるかをまとめて入力する。

自チーム選手は名前で表示する。
相手チーム選手は背番号で表示する。

わからない場合は「？」で進める。

アプリは得点、アウト数、ランナー、打順、投手球数を更新する。
矛盾や不明は要確認として残す。

試合後、閲覧画面でスコアブック風に見直せる。
PDF出力とCSVバックアップができる。
```

## 28. Codexへの注意

まずは見た目よりも、以下を優先して実装する。

- イベントログ設計
- 状態再計算
- 「？」入力の許容
- 矛盾をエラーにしない設計
- 試合中に迷わないシンプルUI

早稲田式スコアブックの見た目は、MVPでは簡易版でよい。
